# Agent 架构设计质量门禁参考

## 门禁等级说明

| 等级 | 含义 | 处理方式 |
|------|------|---------|
| 🚫 BLOCKER | 致命缺陷 | **必须修复**，不通过则禁止进入下一阶段 |
| ⚠️ HIGH | 严重隐患 | 强烈建议修复，上线前必须确认 |
| 📋 MEDIUM | 一般建议 | 可选择性修复，记录在案 |

---

## 全部门禁清单

### 🚫 BLOCKER 级（所有场景通用）

#### G1: 目标定义
- **检查项**：Agent 系统的最终目标和成功标准是否明确
- **通过标准**：描述 ≥15 字，包含"谁"+"做什么"+"产出什么"
- **常见失败原因**：目标过于模糊（如"让 AI 帮我工作"）

#### G2: 退出条件
- **检查项**：是否定义了任务完成的判断标准和终止条件
- **通过标准**：同时包含正常完成标志 + 超时处理 + 死循环防护
- **常见失败原因**：没有定义"什么时候停下来"

#### G3: 异常处理
- **检查项**：是否为每个关键 Agent 定义了失败策略
- **通过标准**：每个 Agent 都有明确的重试/回退/告警方案
- **常见失败原因**：只想着正常路径，没考虑出错怎么办

---

### 🚫 BLOCKER 级（交易执行型专用）

#### G4: 授权机制
- **检查项**：交易账户的授权范围和限额是否明确
- **通过标准**：单笔额度 + 日额度 + 授权有效期均已定义
- **常见失败原因**：直接上手交易，没有限额控制

#### G5: 亏损限制
- **检查项**：是否设置了硬止损线
- **通过标准**：单笔最大亏损比例 + 日亏损熔断线已明确
- **常见失败原因**：没有预设亏损上限

---

### ⚠️ HIGH 级

#### G6: 数据源可靠性
- **检查项**：数据来源是否经过可用性验证
- **通过标准**：确认无反爬、有 rate limit、至少一个备用源

#### G7: 审计日志
- **检查项**：交易类操作是否记录完整日志
- **通过标准**：记录时间、agent、操作内容、结果、耗时

#### G8: 成本估算
- **检查项**：是否估算了 token 消耗和 API 成本
- **通过标准**：低(<10万token)/中(<50万)/高(>50万) 三档至少确认一档

#### G9: 隐私合规
- **检查项**：涉及用户数据时是否有脱敏和合规方案
- **通过标准**：明确 PII 数据类型 + 脱敏规则 + 合规依据

#### G10: 推送通道（报告型）
- **检查项**：报告推送是否定义了主备通道
- **通过标准**：主通道 + 备用通道 + 失败通知策略

---

### 📋 MEDIUM 级

#### G11: 频率限制（数据型）
- **检查项**：数据采集是否设置了合理的频率上限
- **通过标准**：考虑目标服务器限制和成本

---

## 修复模板

当某项门禁未通过时，按以下模板填写修复方案：

```markdown
## [门禁名称] 修复记录

**问题描述**：[具体缺失内容]

**修复方案**：[具体措施]

**验证方式**：[如何确认已修复]

**修复人**：[xxx]  **日期**：[YYYY-MM-DD]
```
