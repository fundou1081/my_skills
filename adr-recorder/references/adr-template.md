## 各部分写作指南

### 背景 (Context)

**好的背景描述应该包含**：
1. **现状**：现在的架构/流程是怎样的
2. **痛点**：存在什么问题，为什么需要改变
3. **驱动因素**：业务需求、技术债务、性能问题等
4. **约束**：时间、资源、技术限制

**示例**：
> 当前系统使用单体架构部署，所有模块耦合在同一进程中。随着用户量增长（已达 500 万日活），部署时间超过 30 分钟，单点故障影响整个系统。新功能开发需要协调多个团队，并发开发效率低下。

### 决策 (Decision)

**决策部分应该清晰回答**：
1. **决定是什么？** - 具体的技术选择或架构变更
2. **为什么选这个？** - 符合业务目标和技术战略
3. **为什么排除其他的？** - 记录被否决的方案和原因

**常见决策类型**：
- 技术栈选择（语言、框架、数据库）
- 架构模式（微服务、模块化、事件驱动）
- 基础设施决策（云服务、部署方式）
- 开发流程（代码规范、审查流程）

### 后果 (Consequences)

**需要考虑的维度**：
1. **技术后果**：性能、可扩展性、可维护性
2. **团队后果**：学习曲线、工作流程变化
3. **业务后果**：交付速度、成本、风险
4. **操作后果**：监控、运维、故障处理

### 合规性 (Compliance)

**确保决策落地的机制**：
1. **自动化检查**：CI/CD 中的验证
2. **人工审查**：代码审查清单
3. **文档约束**：架构规范文档
4. **监控告警**：违反决策时的告警

## ADR 示例

### 示例：采用 TypeScript

```markdown
# ADR-001: 采用 TypeScript 作为前端开发语言

## 状态

> **状态**: Accepted
> **日期**: 2026-03-30

## 1. 背景

### 当前状态
项目使用 JavaScript (ES6+) 进行前端开发，无类型检查。

### 问题陈述
1. 运行时类型错误频发，线上 Bug 率高
2. 大型重构风险高，不敢轻易修改
3. IDE 支持有限，开发效率低
4. 新成员上手周期长，需要大量 Code Review

### 约束条件
- 需要兼容现有 JavaScript 代码
- 团队学习成本需控制在可接受范围
- 不能显著增加构建时间

---

## 2. 决策

### 核心决策
采用 TypeScript 作为主要开发语言，现有 JavaScript 代码逐步迁移。

### 决策理由
1. **类型安全**：编译时发现类型错误，减少运行时 Bug
2. **开发体验**：IDE 智能提示、重构支持
3. **代码文档**：类型即文档，降低学习成本
4. **生态成熟**：主流框架（React、Vue）原生支持

### 考虑的替代方案

#### 方案 A: Flow
- **描述**: Facebook 出品的类型检查器
- **劣势**: 社区支持下降，迁移成本高
- **未选择原因**: 生态萎缩，团队已有 TypeScript 经验

#### 方案 B: 保持 JavaScript + JSDoc
- **描述**: 使用 JSDoc 注释添加类型信息
- **劣势**: IDE 支持不完整，类型推断有限
- **未选择原因**: 无法获得完整类型检查能力

---

## 3. 后果

### 积极影响
- 类型错误减少 70%（基于试点项目数据）
- 重构效率提升，新功能开发速度提升 30%
- 新成员上手时间从 2 周缩短到 1 周

### 消极影响 / 风险
- 团队需要时间学习 TypeScript（预计 2 周）
- 构建时间略有增加（约 10%）
- 部分第三方库类型定义不完整

### 需要的变更
- **配置**: 添加 tsconfig.json、修改构建脚本
- **类型定义**: 为常用库安装 @types 包
- **迁移**: 按模块逐步迁移现有代码

### 技术债务
- 部分老代码需在后续迭代中补充类型
- 计划在 6 个月内完成 80% 代码的 TypeScript 化

---

## 4. 合规性

### 验证方式
- CI 中运行 `tsc --noEmit` 检查类型错误
- Code Coverage 工具统计类型覆盖度

### 代码审查检查项
- [ ] 新增代码必须使用 TypeScript
- [ ] 公开 API 必须有完整类型定义
- [ ] 禁止使用 `any` 类型，特殊情况需注释说明
- [ ] 优先使用 interface 而非 type

### 相关标准
- Google TypeScript 风格指南
- 项目内部 TypeScript 规范文档

### 与其他 ADR 的关联
- **依赖**: ADR-002 (采用 ESLint + Prettier)
```
