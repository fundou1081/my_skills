## 1. TIDD-EC (默认, 精确任务首选)

**全称**: Task, Instructions, Details, Data, Examples, Constraints  
**要素数**: 6  
**默认选择条件**: 任务对准确性敏感 / 数据分析 / 结构化输出

### 6 要素详解

| 要素 | 含义 | 引导问题 |
|---|---|---|
| **T** (Task) | 一句话说清最终目标 | "你想达成什么?" |
| **I** (Instructions) | 详细执行步骤 | "具体要 AI 怎么一步步做?" |
| **D** (Details) | 目标受众、语气、长度等背景 | "谁会读这份输出? 风格要正式还是口语?" |
| **D** (Data) | 原始文本/数据, 让 AI 基于事实 | "有没有原始材料要喂给 AI?" |
| **E** (Examples) | 给 1-3 个范例, 防止幻觉 | "能给我一个你期望的输出样例吗?" |
| **C** (Constraints) | 字数、禁用词、必含项等硬约束 | "输出有什么硬性要求? (字数/格式/必含项/禁用词)" |

### 为什么在数据分析/抽取任务上突出
- **D (Details) 和 D (Data) 分开**: 强制区分"任务语境"和"原始材料", 这一步是防幻觉的关键
- **E (Examples) 显式化**: Few-shot 是抑制幻觉最有效的手段, TIDD-EC 直接列为必填
- **C (Constraints) 收口**: 数据分析对"必须输出 JSON / 必须给引用" 类硬约束要求高, TIDD-EC 把约束放到最后强收

### 适用场景
- 数据分析报告
- 实体识别 / 信息抽取
- 结构化数据处理
- 需要引用的研究类任务
- 代码审查 (要求逐条给证据)

### 不适用场景
- 创意写作 (用 CO-STAR)
- 探索性讨论 (用 BROKE)
- 营销文案 (用 CO-STAR)

### 范例 prompt
```
任务: 分析这批客户评论的情感倾向, 输出正/负/中性比例。

指令:
1. 逐条阅读评论
2. 判断每条情感 (positive/negative/neutral)
3. 统计三类数量, 计算占比
4. 给出 3 条最极端的评论作为佐证

细节: 受众=产品经理, 语气=客观, 长度=200 字以内

资料: [评论数据]

范例:
- 评论: "产品不错, 但物流太慢" → 情感: neutral (产品好, 物流差)
- 评论: "非常满意, 还会回购" → 情感: positive

限制:
- 必须输出 JSON: {"positive": N, "negative": N, "neutral": N, "examples": [...]}
- 禁用词: "非常"、"极其"、"相当"
```

## 2. ICDO (简洁, 一次性任务)

**全称**: Instruction, Context, Constraints, Output  
**要素数**: 4  
**默认选择条件**: 任务简单, 4 要素够用

### 4 要素详解

| 要素 | 含义 | 引导问题 |
|---|---|---|
| **I** (Instruction) | 明确希望 AI 执行的任务 | "具体要做什么?" |
| **C** (Context) | 提供任务所需的背景信息 | "有什么需要 AI 知道的背景?" |
| **C** (Constraints) | 设定格式、风格等限制条件 | "输出有什么限制?" |
| **O** (Output) | 定义输出示例或具体要求 | "你希望输出长什么样?" |

### 适用场景
- 一次性简单任务
- 用户已想清楚, 只需结构化
- 时间紧, 不需要多轮引导

### 范例 prompt
```
指令: 翻译下面的英文邮件成中文, 保留专业语气。

背景: 这是发给客户的合同确认邮件, 客户是德国人, 邮件内容涉及付款条款。

约束:
- 保留"INVOICE"、"PAYMENT TERMS" 等专业术语英文
- 用正式商务中文
- 邮件格式 (开头称呼 / 正文 / 落款)

输出: 一封完整的中文邮件, 可直接发送。
```

## 3. CRISPE (复杂, 需要反问)

**全称**: Capacity/Role, Request/Insight, Statement/Intention, Personality/Style, Experiment, Output Format  
**要素数**: 6  
**默认选择条件**: 用户没想清楚 / 复杂多轮 / 需要 AI 反问

### 6 要素详解

| 要素 | 含义 | 引导问题 |
|---|---|---|
| **C** (Capacity) | 赋予 AI 特定身份 | "AI 应该扮演什么角色? (资深工程师/咨询顾问/...)" |
| **R** (Insight) | 提供任务相关的背景或数据 | "有什么背景信息?" |
| **S** (Statement) | 明确陈述意图 | "具体想做什么?" |
| **P** (Personality) | 指定写作或回复风格 | "风格要专业/幽默/简洁?" |
| **E** (Experiment) | **允许 AI 提问以澄清需求** | "AI 觉得不清楚可以反问" |
| **O** (Output) | 明确最终产出形式 | "最终输出长什么样?" |

### 适用场景
- 复杂多步骤任务
- 用户自己也说"我没想清楚"
- 需要 AI 在过程中不断反问、确认

### 与 TIDD-EC 的区别
- CRISPE **强调 AI 的反问能力** (E), 适合"边做边想"
- TIDD-EC 强调"一次性给齐", 适合"想清楚再干"
- CRISPE 的 C (角色) 比 TIDD-EC 的 D (细节) 更强调 AI 身份

### 范例 prompt
```
角色: 你是一位有 10 年经验的投资顾问。

洞察: 当前市场波动加大, 用户刚收到一笔 50 万的年终奖, 想知道怎么配置。

意图: 给出资产配置建议, 但**先问我几个关键问题再回答**:
- 你的风险偏好?
- 投资期限?
- 现有资产情况?

风格: 客观、专业、不带销售倾向。

输出: 配置建议 + 风险提示, 格式: Markdown。
```

## 4. BROKE (开放, 需要迭代)

**全称**: Background, Role, Objectives, Key Results, Evolve  
**要素数**: 5  
**默认选择条件**: 任务边界模糊 / 探索性 / 需要多轮迭代

### 5 要素详解

| 要素 | 含义 | 引导问题 |
|---|---|---|
| **B** (Background) | 描述任务背景 | "为什么要做这件事?" |
| **R** (Role) | 指定 AI 扮演的角色 | "AI 应该扮演什么角色?" |
| **O** (Objectives) | 明确希望达成的具体目标 | "成功标准是什么?" |
| **K** (Key Results) | 定义期望的输出关键点 | "具体要看到什么产出?" |
| **E** (Evolve) | **根据输出迭代优化** | "我们多轮迭代, 每次改进" |

### 适用场景
- 探索性研究 (用户不知道最终答案是什么)
- 创意发散
- 边界模糊的新需求
- 用户希望"先试试看, 再调整"

### 范例 prompt
```
背景: 公司刚完成 A 轮融资, 准备做品牌升级, 但不知道怎么做。

角色: 你是一位品牌策略顾问, 熟悉初创公司品牌建设。

目标: 3 个月内建立清晰的品牌识别, 触达目标客户。

关键结果:
- 品牌定位 (一句话)
- 视觉规范 (颜色/字体/logo 方向)
- 第一波传播内容主题

试验: 我们分 3 轮迭代:
1. 第一轮: 你给 3 个品牌定位方向
2. 用户选 1 个, 你深化
3. 第三轮: 形成完整品牌手册
```

## 5. CO-STAR (商业, 营销/文案)

**全称**: Context, Objective, Style, Tone, Audience, Response Format  
**要素数**: 6  
**默认选择条件**: 输出面向受众 (文案/营销/演讲/PPT)

### 6 要素详解

| 要素 | 含义 | 引导问题 |
|---|---|---|
| **C** (Context) | 提供背景 | "任务背景是什么?" |
| **O** (Objective) | 说明任务目的 | "想达成什么?" |
| **S** (Style) | 指定写作风格 | "风格要正式/诙谐/学术?" |
| **T** (Tone) | 设定情感基调 | "情感基调是积极/中立/严肃?" |
| **A** (Audience) | 明确目标读者 | "谁会读这个?" |
| **R** (Response) | 定义输出的详细结构 | "输出格式和长度?" |

### 适用场景
- 商业文案 / 营销邮件
- 公开发布的演讲
- 公众号文章
- 任何**面向最终读者**的内容

### 范例 prompt
```
背景: 公司新发布一款面向 Z 世代的 AI 学习产品, 想做上市预热。

目标: 让目标用户对产品产生兴趣, 引导预约试用。

风格: 年轻、有活力、不说教。

语调: 好奇、乐观, 像朋友推荐。

受众: 18-25 岁大学生, 喜欢尝试新科技。

回复格式:
- 标题 (15 字以内, 抓眼球)
- 开头 3 句话 (制造共鸣)
- 3 个 bullet 卖点
- 结尾 CTA (引导试用)
- 总字数 300 字内
```

## 决策树

```
用户给任务
   │
   ├─ Q1: 任务对准确性敏感?
   │      (数据分析/报告/抽取/审查)
   │      └─ 是 ──→ TIDD-EC
   │
   ├─ Q2: 任务边界模糊, 需多轮迭代?
   │      (探索/创意/方向未定)
   │      └─ 是 ──→ BROKE
   │
   ├─ Q3: 用户明确说"我也没想清楚"?
   │      └─ 是 ──→ CRISPE (让 AI 反问)
   │
   ├─ Q4: 输出面向最终受众?
   │      (文案/营销/演讲/PPT)
   │      └─ 是 ──→ CO-STAR
   │
   ├─ Q5: 任务简单, 4 要素够用?
   │      └─ 是 ──→ ICDO
   │
   └─ 默认 ──→ TIDD-EC
```

### 决策树判断顺序 (按优先级)
1. **准确性敏感** > 其他 (TIDD-EC 最强防幻觉)
2. **边界模糊** > **没想清楚** (BROKE 比 CRISPE 更主动引导)
3. **面向受众** > **任务简单** (CO-STAR 必带受众维度)
4. 最后才是 ICDO (兜底)

### 决策冲突时
- **TIDD-EC vs CO-STAR** (都是 6 要素) → 输出是否要"事实准确"? 是 TIDD-EC, 否 CO-STAR
- **BROKE vs CRISPE** (都强调多轮) → 用户想主导还是 AI 主导? 用户主导 BROKE, AI 主导 CRISPE

## 通用万能公式

**用于 Step 5 生成 prompt 时, 作为兜底模板**:

```
请作为 [角色],
根据 [背景信息],
帮我完成 [具体任务]。
要求满足 [约束条件 1]、[约束条件 2]。
最终以 [输出格式] 呈现。
```

**填空示例**:
- 角色: "10 年经验的数据分析师"
- 背景: "我们公司上季度的销售数据"
- 任务: "分析各区域的销售表现, 找出 Top 3 增长点和 Top 3 问题"
- 约束 1: "必须用图表呈现"
- 约束 2: "总字数不超过 500 字"
- 输出格式: "Markdown 报告"

### 何时用
- 5 框架都不太适合时 (极简单任务)
- Step 5 兜底 (用户没选框架)
- 跨框架元素混用时 (例如 TIDD-EC + CO-STAR 的混合体)
