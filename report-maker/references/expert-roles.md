# 多专家角色生成指南

## 概述

多专家视角分析是报告制作的核心方法。通过生成不同领域的专家角色，从多角度拆解问题，获得全面、深入的洞察。

---

## 专家生成原则

### 1. 领域相关性

专家必须与报告主题高度相关：

```
✅ 好的专家选择：
主题：新能源车市场分析
专家：汽车工程师、能源政策研究员、消费者心理学家、供应链专家、充电基础设施规划师

❌ 不好的专家选择：
主题：新能源车市场分析
专家：医学专家、农业专家、文学评论家（无关领域）
```

### 2. 视角多样性

覆盖不同立场、方法论、应用场景：

```
维度分解：
- 技术视角 vs 商业视角
- 宏观视角 vs 微观视角
- 供给方视角 vs 需求方视角
- 短期视角 vs 长期视角
- 风险视角 vs 机会视角
```

### 3. 能力互补性

专家之间能力互补，形成完整分析链：

```
示例组合：
技术架构师 → 技术可行性
产品经理 → 用户需求
财务分析师 → 成本效益
风险控制官 → 风险评估
实施专家 → 落地执行
```

## 专家生成流程

### Step 1: 问题分解

将报告主题分解为核心维度：

```
主题：企业数字化转型战略

核心维度：
1. 技术可行性
2. 业务价值
3. 组织能力
4. 实施路径
5. 风险管控
```

### Step 2: 映射专家

将维度映射到专家角色：

```
技术可行性 → 技术架构师 + 数据架构师
业务价值 → 战略顾问 + 业务分析师
组织能力 → 组织发展专家 + 变革管理顾问
实施路径 → 项目管理专家 + 实施顾问
风险管控 → 风险控制官 + 合规顾问
```

### Step 3: 筛选优化

根据调研深度筛选专家数量：

```
L1 快速概览 → 3个核心专家
L2 标准调研 → 4-5个专家
L3 深度调研 → 5-6个专家
L4 专家级 → 6-7个专家
```

### Step 4: 角色定义

为每个专家定义具体分析任务：

```markdown
【技术架构师】
- 分析现有技术架构的成熟度
- 评估转型的技术可行性
- 提出技术路线建议
- 识别技术风险和障碍

【战略顾问】
- 分析数字化转型的战略价值
- 评估与业务战略的契合度
- 提出战略优先级建议
- 制定价值实现路径
```

## 专家协同模式

### 顺序分析模式

```
专家A（定义问题）
    ↓
专家B（分析原因）
    ↓
专家C（评估影响）
    ↓
专家D（提出方案）
```

### 并行分析模式

```
        ┌→ 专家A（技术视角）
        │
问题 ───┼→ 专家B（商业视角）
        │
        └→ 专家C（风险视角）
                ↓
            综合整合
```

### 交叉验证模式

```
专家A ←→ 专家B：技术可行性 vs 商业价值
专家B ←→ 专家C：商业价值 vs 风险控制
专家C ←→ 专家A：风险控制 vs 技术可行性
```

---

## 注意事项

1. **避免同质化**：专家之间应有明显区分
2. **控制数量**：一般3-7个，过多会分散焦点
3. **动态调整**：根据分析进展可能需要增加新专家
4. **观点冲突**：不同专家可能观点冲突，这反而能发现问题
5. **综合判断**：最终报告需要整合各方观点，给出综合判断
